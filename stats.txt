Current Benchmark Measurements

This file summarizes what the benchmark runs in this repo are actually measuring now.

What each result JSON currently contains

Each benchmark result file has two measurement layers:

1. Per-sample stats
Stored under `report.results[i].stats`

2. Per-run summary stats
Stored under `report.stats`

The comparison report reads the run-level stats and writes telemetry summaries into `comparison/output`.

What is currently measured directly

Runtime

- Wall-clock time per sample
Measured around the whole sample path, including image loading, prompt construction, model inference, parsing, and evaluation.

- Total generation time
Measured around the `model.predict(...)` call only.

- Model load time
Measured once when the model object is created before the benchmark loop starts.

- Samples per second
Computed from total run wall-clock time and number of completed samples.

- Time per output token
Computed when output token count is available.

- Tokens per second
Computed from output token count divided by generation time when both are available.

Work done

- Number of output tokens
Counted with the model tokenizer when the model exposes one through the processor or tokenizer.

- Number of generated outputs
For most tasks this is `1` if there is a non-empty response, otherwise `0`.
For detection tasks this is the number of parsed predicted boxes.

- Number of benchmark samples completed
Count of samples whose model call finished without raising an exception.

- Success count / failure count
A sample is marked failed if `model.predict(...)` raises an exception.

- Retry count
Present in the schema, but currently always `0`.
The runner does not yet implement retry logic.

Hardware / resource use

- Peak CPU RAM
Measured with `psutil` while the sample is running.

- CPU utilization
Measured with `psutil` as a sampled process-level average while the sample is running.

- Peak GPU memory
Measured only if CUDA is available through `torch.cuda`.
On this machine the field is currently usually `null`.

What is currently inferred or approximated from model output

Model behavior

- Truncated
Set to true when counted output tokens are greater than or equal to `model.max_new_tokens`.
This is a heuristic, not a direct generation-stop reason from the model backend.

- Empty response
True when the prediction string is empty after stripping whitespace.

- Invalid-format output
Task-specific heuristic.
For classification-style tasks it means the output does not match an allowed prompt label.
For multiple-choice tasks it means the selected answer did not resolve cleanly to one of the choices.
For captioning and VQA it is mostly a simple format check such as multi-line output.
For detection it is treated as invalid when there is text but no parseable detection boxes.

- Parse failure
Task-specific heuristic.
Most meaningful for detection and multiple-choice tasks.

- Hallucinated-label count
Currently heuristic.
For classification it is derived from invalid label output.
For detection it counts parsed labels that are outside the allowed label list.

- False positive count / false negative count
Currently real for detection tasks, taken from the detection evaluator.
For non-detection tasks they remain `0`.

- Predicted detection count
For detection tasks this is the number of parsed predicted boxes.
For other tasks it is usually `0`.

What benchmark quality metrics are currently measured

These are stored in the result payload itself and are the values used by the comparison report.

- Accuracy
Used for labeling and QA-style comparisons.

- BLEU
Used for captioning comparisons.

- Precision / recall / F1
Used for detection comparisons.

- IoU
Stored for detection matching analysis.

What the comparison report currently measures

The report generator now produces two kinds of outputs:

1. Score comparison outputs
Files such as:
- `comparison/output/model_summary.csv`
- `comparison/output/family_summary.csv`
- `comparison/output/family_radar.png`

These are based on benchmark quality only:
- accuracy
- BLEU
- detection F1

2. Telemetry comparison outputs
Files such as:
- `comparison/output/telemetry_model_summary.csv`
- `comparison/output/telemetry_by_benchmark.csv`
- `comparison/output/telemetry_mean_wall_clock_seconds.png`
- `comparison/output/telemetry_mean_generation_seconds.png`
- `comparison/output/telemetry_mean_tokens_per_second.png`
- `comparison/output/telemetry_peak_cpu_ram_gib.png`
- `comparison/output/telemetry_coverage.png`

These are based on `report.stats`.

What is present in the schema but not truly measured yet

- First-token latency
Currently always `null`.
No streaming callback timing is implemented.

- GPU utilization
Currently always `null`.
The sampler does not read utilization from NVML or `nvidia-smi`.

- VRAM allocation over time
Currently always `null`.
Only peak CUDA allocation is tracked.

- Disk offload volume
Currently always `null`.
No reliable offload byte counter is implemented.

- Retry behavior
Schema exists, but there is no actual retry loop in the benchmark runner.

Important caveats

- Some `qwen25-vl` telemetry in the current `results/` directory was backfilled from legacy result files.
Those rows are explicitly tagged with:
`stats_source = estimated_from_legacy_result`
`stats_measured = false`

- Comparison plots for time and RAM mix measured and estimated telemetry unless you filter by `stats_measured`.

- Detection quality is still coming from the saved benchmark result files.
If detection scores are all zero in the JSON, the comparison plots will correctly show detection as zero.

Short version

The repo is currently measuring:
- real per-sample wall-clock time
- real generation time
- real model load time
- real peak CPU RAM
- real process CPU usage
- token-count-based throughput when token counting works
- task quality metrics like accuracy, BLEU, and detection F1

The repo is not yet truly measuring:
- first-token latency
- GPU utilization
- VRAM time series
- disk offload volume
- real retry behavior
